# 后续开发路线：创作者素材库与重写工作台

## 1. 定位

当前项目已经从“知乎收藏夹导出器”演进成一个本地知识库工具。后续不建议继续围绕“导出”扩展功能，而应围绕内容创作者的实际工作流建设：

> 把收藏内容变成可检索、可聚类、可重组的个人素材库，再辅助生成更像自己的文章、回答和播客大纲。

个人使用阶段可以服务“洗稿”和素材重组，但如果后续推广，产品表述应转为“素材消化”“观点重构”“二次创作辅助”。功能上需要保留来源追踪、相似度提醒和引用提示，降低版权与平台风险。

## 2. 产品原则

- **服务写作结果**：词云和知识图谱不是展示功能，而是选题、拆稿、重写、扩写的入口。
- **保留来源链路**：任何生成内容都应能追溯到参考素材，避免素材被黑盒化。
- **本地优先**：Cookie、素材库、向量库、AI Key 默认保存在本机，降低隐私和部署复杂度。
- **先个人效率，再商业包装**：先验证个人工作流是否高频有效，再考虑多用户、计费和同步。
- **避免“洗稿”产品化表述**：对外推广时强调原创辅助、素材管理和观点重构。

## 3. 目标工作流

理想使用路径：

1. 从知乎、Bilibili、小宇宙等来源导入收藏或内容。
2. 系统自动抽取素材卡片，包括核心观点、论据、案例、金句和反方观点。
3. 系统按主题聚类，生成选题地图。
4. 用户输入一个题目，系统返回写作素材包。
5. 用户选择角度、结构和素材，生成初稿。
6. 系统检查与原素材的相似度，并标出高风险段落。
7. 输出 Markdown，同时记录 `derived_from` 来源关系。

## 4. 词云升级方向

当前词云只展示词频，容易停留在“AI、产品、用户”这类低价值结果。后续应升级成“主题雷达”。

### 4.1 关键词词云

基础层仍保留词频展示，但需要支持：

- 按平台过滤：知乎 / Bilibili / 小宇宙
- 按收藏夹过滤
- 按时间范围过滤
- 按作者过滤
- 点击关键词进入语义搜索

技术上可继续用 Python `wordcloud` 生成静态图，也可以迁移到 ECharts word cloud 或 `wordcloud2.js`，让关键词可点击、可筛选。

### 4.2 主题簇词云

比词频更重要的是主题聚类。可以基于 ChromaDB 中的 embedding，对 chunk 或内容做聚类，生成 20-50 个主题簇。

每个主题簇应包含：

- 主题标题
- 代表关键词
- 代表素材
- 涉及平台和收藏夹
- 最近更新时间
- 素材数量

示例主题：

- AI 编程 / Agent / Cursor / 工作流
- 内容创作 / 选题 / 爆款结构
- 产品增长 / 用户洞察 / 商业模式

### 4.3 写作机会词云

最终词云应回答“什么主题值得写”，而不是“什么词出现最多”。

建议给每个主题打分：

- 素材数量
- 最近热度
- 观点冲突度
- 高质量素材占比
- 是否已有输出
- 是否适合知乎回答、公众号文章或播客大纲

输出可以是“写作机会榜”：

| 主题 | 素材数 | 冲突度 | 新鲜度 | 推荐输出 |
|---|---:|---:|---:|---|
| AI 编程工作流 | 42 | 高 | 高 | 知乎回答 |
| 内容创作方法论 | 31 | 中 | 中 | 公众号文章 |
| 独立开发商业化 | 18 | 高 | 高 | 播客大纲 |

## 5. 知识图谱升级方向

当前图谱如果只连接“文章 - 作者 - 收藏夹”，很快会变成一团线。后续应从文件关系图升级为观点关系图。

可参考方向：

- Logseq / Roam Research：双链和块级引用
- Obsidian Graph View：本地知识库关系展示
- Microsoft GraphRAG：实体、关系、社区摘要
- Neo4j Bloom：可探索的实体关系图
- Cytoscape.js / Sigma.js：前端交互图谱

### 5.1 节点类型

建议图谱节点从当前的 content / author / collection 扩展为：

- `Content`：原始文章、回答、视频、播客
- `Author`：作者
- `Topic`：主题
- `Claim`：核心观点
- `Evidence`：案例、数据、故事
- `Counterpoint`：反方观点
- `Output`：用户生成的文章、回答、播客稿

### 5.2 边关系

建议支持以下关系：

- `belongs_to`：内容属于某收藏夹
- `by_author`：内容来自某作者
- `about`：内容涉及某主题
- `supports`：论据支持某观点
- `contradicts`：观点之间存在冲突
- `similar_to`：内容相似
- `derived_from`：输出参考了哪些素材

这类图谱可以回答实际写作问题：

- “我要写 AI 取代程序员，有哪些支持观点？”
- “有哪些反方观点可以增加文章厚度？”
- “哪些案例可以作为开头？”
- “这篇输出参考了哪些原始内容？”

### 5.3 图谱交互

图谱 UI 不应一次展示全部节点。推荐交互方式：

- 默认展示主题簇，而不是全部内容。
- 点击主题后展开相关观点和素材。
- 点击观点后显示支持素材、反方观点和可引用片段。
- 点击输出文章后显示 `derived_from` 来源链。
- 支持按平台、作者、收藏夹、时间过滤。

## 6. 核心功能路线

### 阶段 0：稳定基础

目标：先让现有同步和知识库可信。

任务：

- 修复知识库删除 Markdown 后旧 chunk 未清理的问题。
- 修复知乎同步中 `updated_time` 与 `collect_time` 混用的问题。
- 修复 Bilibili 同步只要已有 hash 就永久跳过的问题。
- 给项目补 `.gitignore`，排除 `.venv`、`__pycache__`、`.pyc`、`egg-info`、`.pytest_cache`。
- 补最小测试集：配置、Markdown 转换、chunk、知识库增量删除、搜索筛选。

验收：

- 重复构建知识库不会产生重复向量。
- 删除源 Markdown 后搜索不到旧内容。
- 内容更新时间变化后同步能更新。
- `pytest` 至少覆盖核心数据流。

### 阶段 1：素材卡片

目标：把原始内容转换为可复用的写作素材。

每篇内容生成一张素材卡片：

- 标题
- 来源 URL
- 作者
- 平台
- 核心观点
- 关键论据
- 案例 / 故事
- 金句
- 反方观点
- 适合写作角度
- 风险提示

实现建议：

- 先用规则 + LLM 生成 JSON。
- JSON 存到 Markdown frontmatter 或 `.knowledge/cards/*.json`。
- 卡片与原文保持 `content_id` 关联。

### 阶段 2：主题聚类与主题页

目标：让用户看到素材库里真正的主题结构。

主题页应包含：

- 主题摘要
- 关键词
- 代表素材
- 主要观点
- 反方观点
- 可写选题
- 相关主题

实现建议：

- 用 embedding 聚类生成主题簇。
- 先用 KMeans 固定主题数，后续再尝试 HDBSCAN。
- 主题标题可由 LLM 根据代表 chunk 自动生成。

### 阶段 3：写作素材包

目标：输入一个题目后，不直接生成文章，而是先生成可控素材包。

素材包结构：

- 题目理解
- 推荐切入角度
- 可用观点
- 可用论据
- 案例
- 反方观点
- 推荐结构
- 引用来源
- 高风险相似片段

这一步是个人“洗稿”工作流的核心。它让用户先判断素材是否足够，再决定是否生成初稿。

### 阶段 4：生成与相似度检查

目标：降低直接照搬风险，提高输出可控性。

生成后进行检查：

- 与原文 chunk 的相似度
- 高相似段落标注
- 建议改写方向
- 来源覆盖情况
- 是否过度依赖单一来源

推荐策略：

- 对生成文本按段落切分。
- 与素材库 chunk 做 embedding 相似度检索。
- 高于阈值的段落标红。
- 要求用户二次编辑或触发改写。

### 阶段 5：重做词云和知识图谱 UI

目标：把可视化变成创作入口。

词云：

- 从“词频图”升级为“主题机会图”。
- 点击主题进入主题页。
- 展示写作机会评分。

知识图谱：

- 从 Pyvis 迁移到 Cytoscape.js 或 Sigma.js。
- 默认展示主题网络。
- 支持逐层展开观点、素材和输出。
- 输出文章显示来源链路。

### 阶段 6：推广准备

目标：从个人工具变成可被他人理解和使用的产品。

建议包装：

- “个人素材库”
- “创作者知识库”
- “选题重构工具”
- “AI 写作素材工作台”

不建议对外使用：

- “洗稿”
- “搬运”
- “仿写爆款”

推广前需要：

- 清晰安装流程
- 示例数据
- Demo 视频
- 本地隐私说明
- API Key 管理说明
- 来源追踪和相似度检查能力

## 7. 技术选型建议

### 7.1 可视化

- 词云：ECharts word cloud 或 `wordcloud2.js`
- 图谱：Cytoscape.js 或 Sigma.js + Graphology
- 统计图：ECharts
- 原型期：Streamlit 继续验证
- 推广期：React + Tauri 或 Electron

### 7.2 存储

短期：

- Markdown 原文
- ChromaDB 向量
- JSON manifest
- `.knowledge/cards/*.json`
- `.knowledge/topics/*.json`

中期：

- SQLite 存内容、卡片、主题、输出、关系边
- ChromaDB 继续做向量检索

长期：

- 如图谱复杂度明显上升，再考虑 Neo4j。

### 7.3 AI 流程

建议从“直接写作”改成分阶段：

1. 抽取素材卡片
2. 聚类主题
3. 生成写作素材包
4. 生成初稿
5. 相似度检查
6. 改写建议
7. 输出归档

这样更可控，也更适合后续推广。

## 8. 对外产品形态

### 8.1 个人版

本地 CLI + 本地 Web UI。

特点：

- 本地存储
- 用户自带 API Key
- 适合个人创作者
- 安装成本较高，但隐私压力小

### 8.2 Pro 本地应用

Tauri / Electron 打包。

特点：

- 安装体验更好
- 可内置同步任务
- 可做图谱和主题页
- 适合小范围推广

### 8.3 SaaS 版本

不建议早期做。SaaS 会遇到：

- Cookie 合规问题
- 内容版权问题
- 用户隐私问题
- 平台反爬风险
- 成本和计费复杂度

只有在本地版验证稳定后，再考虑 SaaS。

## 9. 风险与边界

### 9.1 版权风险

风险：生成内容与原文过于相似。

缓解：

- 来源追踪
- 相似度检查
- 高风险段落提示
- 鼓励观点重构，而不是逐句改写

### 9.2 平台风险

风险：知乎、Bilibili 等平台接口变化或限制访问。

缓解：

- 降低请求频率
- 本地缓存
- 失败重试
- 手动导入备选方案

### 9.3 产品表述风险

风险：“洗稿”定位不适合公开推广。

缓解：

- 对内用“重写工作流”描述即可。
- 对外统一使用“素材消化”“二次创作辅助”“观点重构”。

### 9.4 AI 质量风险

风险：LLM 生成空泛、失真或幻觉。

缓解：

- 先生成素材包，再生成正文。
- 强制引用素材来源。
- 保留用户编辑环节。
- 对事实性内容做来源回链。

## 10. 推荐近期任务清单

优先级从高到低：

1. 修复同步和知识库增量问题。
2. 补项目级 `.gitignore` 和最小测试集。
3. 实现素材卡片抽取。
4. 实现主题聚类和主题页 JSON。
5. 实现写作素材包命令。
6. 实现生成文本相似度检查。
7. 把词云升级为主题机会图。
8. 把知识图谱升级为观点关系图。
9. 再考虑重做 Web UI 技术栈。

## 11. 判断标准

后续功能是否值得做，可以用这几个问题判断：

- 它能否帮助用户更快选题？
- 它能否帮助用户找到可复用素材？
- 它能否让生成内容更像用户自己的表达？
- 它能否降低照搬风险？
- 它能否保留来源链路？

如果答案是否定的，即使视觉效果好，也不应优先做。
